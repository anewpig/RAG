第一層：最值得先補，履歷價值最高
這些是我最推薦你下一階段加的。
1. Evaluation 測試集
自己做 20～50 題問答集，包含：
可回答問題
不可回答問題
跨文件問題
流程型問題
精確資訊問題
然後記錄：
top-1 / top-3 有沒有抓對
是否應該拒答
citation 是否對應正確
這個一加上去，你就能在面試講：
我不只是做出 RAG，我還有做 retrieval quality evaluation。
2. Better chunking
你現在是 token window chunking，之後可以升級成：
section-aware chunking
FAQ 一題一 chunk
SOP 按步驟切
markdown heading-aware chunking
這會讓你能回答：
我怎麼切 chunk，為什麼這樣切。
3. 更好的 citation 顯示
現在 citation 已經夠用了，但之後可以補成：
顯示文件名 + 頁碼 + chunk id
點 citation 展開原文
回答句子對應特定 citation
這會讓整體更像真正的企業知識庫。
4. Retrieval debug dashboard
例如新增一個 debug endpoint 或頁面，顯示：
query
top-k chunks
distance
metadata
最後是否拒答
這個很加分，因為它代表你有在做 observability。
第二層：讓專案更像實務產品
這些是第二波可以補的。
5. Metadata filter
例如只查某份文件、某類文件、某部門文件：
HR
IT
Finance
FAQ / SOP / policy
這樣能展示你知道 retrieval 不是只能靠向量相似度。
6. Hybrid retrieval
把 keyword search + vector search 合在一起。
因為很多精確詞像：
VPN
HR
條號
表單名
日期
其實單靠向量不一定最穩。
7. Reranking
先 retrieve top-k，再用第二階段排序模型重排。
這通常是讓 RAG 品質往上跳的關鍵步驟之一。
8. 真正的 generation
你現在有 fallback answer，之後若要升級，可以加：
OpenAI
Ollama / 本地 LLM
結構化輸出 JSON
這樣回答會比現在自然很多。
第三層：做成作品集等級
這些是你之後想讓它變漂亮、能 demo 給別人看時再補。
9. 前端介面
用 Streamlit 或簡單前端做：
上傳文件
問問題
顯示答案
顯示 citations
顯示 debug 資訊
10. Docker
把專案容器化，之後面試時可以說：
這個專案可以直接用 Docker 跑起來。
11. 單元測試
例如測：
文件清理
chunking 數量
metadata 欄位完整性
retrieval 是否回傳結果
12. README 完整化
README 補成這些區塊：
專案動機
系統架構
chunking 策略
metadata schema
retrieval 流程
citation 設計
hallucination control
evaluation 方法
未來改進方向
